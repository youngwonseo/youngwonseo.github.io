Creating a Continuous Delivery Pipeline

2019-08-17 / 서영원

이번에는 Continous Deliivery Pipeline을 배포하는 방법을 살펴보겠습니다. 이것은 build, stage, test, deploy를 automated triggers나 manual approvals을 사용해서 가능하게 합니다. 이를 위해 Jenkins와 Spinnaker 사용많이 사용합니다.

  • Jenkins가 인기는 더 많음
  • Spinnaker는 Netflix에서 개발, Kubernetes engine, AWS 등에서 클라우드 서비스를 위해 사용

  • Continuous Delivery를 사용하여 build, test, deploy할 응용의 환경입니다.
  • 일반적인 frontend 와 backend가 존재
  • 2개의 service가 존재, 각 서비스는 각각의 pod을 가짐
  • single cluster에서 운영됨

  • 응용 업데이트에 대한 continuously deliver을 제공하기 위해 reliablly build, test, update가 자동화되어야 함
  • code change는 artifact creation, unit testing, functional testing, production rollout이 포함된 pipeline에서 자동적으로 수행되어야 함
  • 특정경우 업데이트 결과를 일부 사용자들에게만 제공하여 운영(실패시 roll back)

Cloud Build, Spinnaker를 사용한 Continous delivery

  • Git tag를 사용해서 app생성, 특정 Git tag를 가진 소스가 변경(push)되면 Cloud Build가 작동(trigger)하도록 설정(Git repository또는 Cloud source repository)
  • Cloud Build에서 produce artifacts(Docker images, unit test 실행, push image to Spinnaker for deployment) 설정
  • Cloud Build는 각 build 단계를 Docker container안에서 실행
  • Cloud Build 또는 custom build steps을 사용가능
  • Continous delibery pipeline의 마지막 단계는 Spinnaker를 사용해서 canary 버전에 배포하는것
  • 이후 production 환경에 자동으로 배포

Jenkins의 Pipeline

  • Jenkins을 사용한 pipeline 구축도 위 환경과 비슷
  • 위 그림의 blue boxes가 Jenkins의 build, Gray boxes는 deploy
  • Dev environment에서 실제 운영환경과 비슷하게 테스트
  • Canary가 verify되면 production에 merge
